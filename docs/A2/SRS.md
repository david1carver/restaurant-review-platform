# A2 Step 2 — Software Requirements Specification (SRS)

**System:** Restaurant Review Platform (extended for IFQ636 Assignment 2)
**Stack:** Node.js/Express, MongoDB/Mongoose, React, JWT auth; deployed via GitHub Actions to AWS EC2 (nginx + pm2).

---

## 2.1 Purpose
The purpose of the system is to provide a trustworthy, peer-driven platform for
discovering restaurants and sharing dining experiences. It is being developed to
replace fragmented, unverified word-of-mouth and scattered social-media opinions
with a single, structured catalogue where diners can browse restaurants, read and
contribute star-rated reviews, and where owners and administrators can curate and
moderate content. For Assignment 2 the original A1 application is extended with an
object-oriented, design-pattern-based architecture and a new in-app notifications
subsystem.

## 2.2 Problem statement
Choosing where to eat is difficult because the volume of options is large and
quality is variable, while existing review signals are noisy, inconsistent, or
gamed (e.g. duplicate or fake reviews). Restaurant owners lack a direct channel to
respond to feedback, and there is no lightweight moderation path for inappropriate
content. The system addresses these problems with: one verified review per diner
per restaurant, automatically maintained aggregate ratings, an owner-response
capability, administrative moderation, and notifications that keep diners informed
when their content changes state.

## 2.3 Scope
**Product overview.** A full-stack web application with a public catalogue, an
authenticated diner experience, and an admin panel.

**In scope (goals & benefits):**
- Account registration, login, and JWT-secured sessions (role: diner / admin).
- Browse, search (by name/cuisine/location), filter (min rating) and sort restaurants.
- Full CRUD on reviews by their author; one review per diner per restaurant.
- Automatically recomputed `averageRating` and `reviewCount` per restaurant.
- Owner/admin response to a review.
- Admin CRUD on restaurants (with cascade delete of reviews) and review moderation.
- In-app notifications generated by domain events (new A2 functionality).
- Automated build/test and deployment via CI/CD.

**Out of scope (exclusions):** real-money payments/reservations, native mobile apps,
direct messaging between users, third-party review imports, multilingual content,
and recommendation/ML ranking.

## 2.4 User characteristics
| Role | Description | Skill level | Needs |
|---|---|---|---|
| Diner (registered user) | General public seeking dining options | Low–medium; everyday web users | Fast browse/search, simple review forms, clear feedback |
| Restaurant owner* | Responds to reviews of their establishment | Medium | Visibility of feedback, ability to respond |
| Administrator | Curates catalogue and moderates content | Medium–high | Efficient CRUD, moderation tools, oversight |
| Guest (unauthenticated) | Reads public catalogue/reviews | Low | Frictionless read-only access |

*In the assignment scope the owner-response action is exercised through the admin role.

**Accessibility considerations:** semantic HTML and labelled form controls; sufficient
colour contrast in the design system; keyboard-navigable forms; responsive layout
usable on mobile and desktop; status conveyed by text + colour (not colour alone).

## 2.5 Constraints
- **Technology:** must extend the nahaQUT starter (Node.js, React, MongoDB); auth via JWT/bcrypt.
- **Hardware/host:** target deployment on an AWS EC2 t2.micro (free-tier) instance.
- **Regulatory/privacy:** passwords stored only as bcrypt hashes; no plaintext secrets in source (GitHub Secrets used); personal data limited to name/email.
- **Project constraints:** individual + team delivery; GitHub-based collaboration; CI/CD via GitHub Actions; 3000-word report limit.
- **Known environmental constraint:** the shared student AWS account restricts EC2 Instance Connect and Elastic IP allocation (see CI/CD runbook), which affects stable public exposure.

## 2.6 Functional requirements
1. The system **shall** allow a user to register with name, email, and password.
2. The system **shall** allow a user to log in with email and password and receive a JWT.
3. The system **shall** restrict admin-only operations to users with the `admin` role.
4. The system **shall** allow anyone to browse restaurants with pagination.
5. The system **shall** allow anyone to search restaurants by name/cuisine/location and filter by minimum rating, and sort by rating/newest/name.
6. The system **shall** allow anyone to view a restaurant's details and its reviews.
7. The system **shall** allow an authenticated diner to create one review (1–5 star rating + text) per restaurant.
8. The system **shall** reject a second review by the same diner for the same restaurant (HTTP 409).
9. The system **shall** allow a review's author to update or delete their own review.
10. The system **shall** allow an administrator to delete any review (moderation).
11. The system **shall** recompute a restaurant's `averageRating` and `reviewCount` after any review create/update/delete.
12. The system **shall** allow an administrator to create, update, and delete restaurants; deleting a restaurant **shall** cascade-delete its reviews.
13. The system **shall** allow an admin/owner to post a response to a review.
14. The system **shall** generate an in-app notification on review publication, owner response, and moderator removal.
15. The system **shall** allow a user to list their notifications, see an unread count, and mark a notification as read.

## 2.7 Non-functional requirements (NFRs)
- **Performance:** restaurant search returns within ~2 s under typical load; reads are paginated (≤50/page).
- **Reliability:** non-critical side-effects (notifications, audit) are isolated so a failure does not break the originating request; backend kept online by pm2 with auto-restart.
- **Security:** bcrypt password hashing; JWT with expiry; role-based authorization middleware; secrets in GitHub Secrets, never in source.
- **Usability:** responsive UI; consistent design system; clear validation/error messages.
- **Scalability:** stateless JWT auth and a layered (repository/service) architecture allow horizontal scaling and storage swap behind the repository interface.
- **Maintainability:** OOP design with design patterns; unit-tested core; CI on every push.

## 2.10 Safety considerations
- **Data protection:** bcrypt-hashed passwords; JWT secret and DB URI injected via
  environment/Secrets; passwords never returned by the API (`.select('-password')`).
- **Error handling & recovery:** a typed exception hierarchy (`AppError` + 400/401/403/404/409)
  is mapped to consistent HTTP responses by a single `sendError` handler; controllers
  never leak stack traces. Non-critical observers are wrapped so failures are logged,
  not propagated. pm2 auto-restarts the backend on crash/reboot; MongoDB Atlas provides
  managed backups.
- **Input safety:** rating bounds (1–5) and required-field validation; regex inputs are
  escaped before use in queries to prevent injection via search terms.

## 2.11 Risk management
| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| EC2 public exposure blocked by student AWS account limits | High | High | Document EC2 target + scripts; provide managed-host fallback; keep test stage automated (see CI/CD runbook) |
| Duplicate/abusive reviews | Medium | Medium | Compound unique index (one review/diner/restaurant → 409); admin moderation |
| Leaked secrets | Low | High | GitHub Secrets, `.env` git-ignored, no secrets in code |
| Rating inconsistency after concurrent edits | Medium | Medium | Server-side recompute via Observer after every mutation |
| Team member non-contribution | Medium | Medium | Branch-per-feature, PR reviews, Part C declaration, commit-author checks |
| Regression during refactor | Medium | High | 44 automated unit tests run in CI on every push |
| Notification side-effect failure breaks core flow | Low | Medium | Observers isolated (try/catch per observer) |
